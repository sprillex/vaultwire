# VaultWire: Architecture Evaluation & Implementation Blueprint

## Part 1: Architecture & Plan Evaluation

### 1. Security
*   **Wireless Attack Surface (Bluetooth):** Relying on a Bluetooth keyboard introduces a significant wireless attack vector. Bluetooth traffic can be intercepted, or the Pi could be subjected to pairing attacks or vulnerabilities like BlueBorne.
    *   *Mitigation:* Replace the Bluetooth keyboard with a hardwired physical keypad or a wired USB keyboard connected directly to the Pi via a USB OTG hub. This enforces a true physical air-gap.
*   **USB Stack Enumeration Exploits:** While HID is unidirectional in application, the underlying USB protocol is bidirectional during device enumeration. A malicious host could attempt fuzzing the Pi's USB stack.
    *   *Mitigation:* Statically compile the HID gadget driver into the kernel and explicitly disable all other USB composite modules to minimize the attack surface.
*   **Host-Side Keylogging:** The Pi ultimately types the decrypted passwords into the host system as plain text via HID. Malware or keyloggers on the host OS will capture these credentials.
    *   *Mitigation:* Acknowledge this inherent limitation in the threat model. Hardware injectors cannot bypass compromised host software.
*   **Sync Mode Integrity & Tampering:** The Pi transmits layout data blindly to the host. A malicious actor on the host could manipulate the local `mint_vault_data.json` to alter the `id` tags, tricking the user into injecting the wrong password.
    *   *Mitigation:* Implement an HMAC (Hash-Based Message Authentication Code) on the sync string generated by the Pi, using a key derived from the master password or established during setup. The companion app must verify this signature to ensure layout integrity.
*   **Cold Boot Attacks & RAM Volatility:** Decrypting the `.kdbx` directly into volatile RAM is good. However, if the Pi is powered off, residual data can persist in memory chips briefly.
    *   *Mitigation:* Ensure secure memory allocation (e.g., using `mlock`) and disable all swap space. Implement a RAM scrubbing routine that overwrites memory before powering down or locking.
*   **Rate-Limiting & Sequential Harvest Traps:** An attacker with physical access to the Pi keyboard could rapidly sequentially type IDs (1, 2, 3...) to dump the database.
    *   *Mitigation:* Implement a hardware lockout period (e.g., a token-bucket rate limiter) between injections. Add a "Honey-Pot Trigger": if sequential lookups or excessive requests are detected, immediately wipe RAM and trigger a cold reboot.
*   **Physical Coercion (Duress / Panic PIN):** An attacker might physically force the user to unlock the vault. Standard digital security cannot protect against physical coercion.
    *   *Mitigation:* Implement a secondary "Panic PIN". If entered, the Pi will seamlessly decrypt a secondary `.kdbx` file filled with realistic dummy accounts (The Fake Vault) to provide plausible deniability.
*   **Isolated Storage Container (Virtual Drive):** Mounting the raw Pi filesystem via `libcomposite` as a Mass Storage Device exposes the OS to host-based malware.
    *   *Mitigation:* Expose only a pre-allocated, flat image file (e.g., `vault.img`) rather than the actual OS partition.
*   **Kernel-Level Execution Blocks:** Malware dropped into the `.kdbx` storage folder could be executed by the Python daemon.
    *   *Mitigation:* Mount the `vault.img` container internally using strict Linux kernel flags: `ro,noexec,nosuid,nodev`.
*   **Physical Tampering (Evil Maid Attack):** An attacker with brief physical access could pull the SD card and replace the OS with a malicious image designed to silently log the master password or exfiltrate the `.kdbx` file over a hidden wireless interface.
    *   *Mitigation:* Address physical security in the enclosure design (see `PHYSICAL_DESIGN.md`). Additionally, implementing an encrypted root filesystem (LUKS) should be considered a stretch goal to protect the underlying OS integrity against offline modifications.
*   **Parser Vulnerabilities:** As `pykeepass` processes external files, it is theoretically vulnerable to memory exhaustion or logic flaws if a maliciously crafted `.kdbx` file is intentionally uploaded during Sync Mode.
    *   *Mitigation:* Ensure `pykeepass` is strictly version-pinned and regularly audited. Include a security step to perform fuzzing on the vault parsing logic.

### 2. Best Practices
*   **Dependency Management & Environments:** Relying on global system packages (`sudo apt install`) and global Python installations (`pip3 install`) can lead to dependency hell or conflicts.
    *   *Recommendation:* Enforce Python Virtual Environments (`venv`) for both the Host and Pi development. Use `requirements.txt` or `pyproject.toml`.
*   **Principle of Least Privilege:** Running the Pi daemon as root to access `/dev/hidg0` violates security best practices.
    *   *Recommendation:* Create a dedicated unprivileged `vault_daemon` user. Use `udev` rules to grant this user write access strictly to `/dev/hidg0`.
*   **Modular Architecture & Plugin Pattern:** The parser (`pykeepass`) and user interface logic are deeply coupled in the existing plan.
    *   *Recommendation:* Abstract the vault parsing logic into a modular plugin interface. This allows future expansion (e.g., parsing Bitwarden exports) without rewriting the core HID engine. Similarly, modularize the host app into distinct files for UI, storage, and sync logic.
*   **State Feedback (Blind Operations):** Currently, entering the Master Password or Target IDs via the dedicated keyboard is completely "blind." If a user makes a typo or the system hangs, there is zero feedback.
    *   *Recommendation:* Adding a physical screen (e.g., a simple OLED) provides essential state feedback. For this development version, it will remain a stretch goal until the final form factor of the case can be selected.
*   **Vault Backup Strategy:** The architecture focuses on reading and updating the vault, but does not explicitly define a backup strategy. SD cards have high failure rates.
    *   *Recommendation:* Define a secure backup procedure utilizing the Sync Mode Mass Storage container, allowing users to easily and safely copy the encrypted `.kdbx` file back to the host system for archiving.

### 3. Practical Applications & Feasibility
*   **Typing Speed Limitations:** Blasting keystrokes via USB HID can lead to dropped characters due to host OS load or USB polling rates.
    *   *Recommendation:* The payload injection script must include micro-delays (e.g., 5-10ms) between simulated keystrokes.
*   **Caps Lock Inversion:** If the host PC has Caps Lock engaged, injected passwords will be case-inverted, causing failed logins.
    *   *Recommendation:* The Pi must force the Caps Lock state off via a targeted HID macro sequence before sending the payload.
*   **Air-Gapped Update Lifecycle (Database Ingress):** Updating the database on a read-only, networkless device is an operational bottleneck.
    *   *Recommendation:* The physical Hardware Maintenance Switch (GPIO) is a highly realistic solution. Position A boots into Vault Mode (read-only, HID), and Position B boots into Sync Mode (read-write, Mass Storage).
*   **Dependency Hell on ARM:** Building cryptography libraries (required by `pykeepass`) directly on a Pi Zero 2 can be extremely slow and prone to failure.
    *   *Recommendation:* Provide a pre-configured Docker container (using buildx) to cross-compile the Pi's Python environment and dependencies from a more powerful host machine before deployment.

## Part 2: Step-by-Step Implementation Blueprint

### Phase 1: Environment & Foundational Architecture
**Objective:** Establish secure, isolated development environments and modular project structures for both the host and Pi systems.
*   **Prerequisites:** Python 3.9+, Git, Docker (for cross-compilation).
*   **Checklist:**
    *   [ ] Initialize the project repository with `host/` and `pi/` directories.
    *   [ ] Create strict `requirements.txt` files for both environments.
    *   [ ] Setup Python virtual environments (`venv`) and document activation steps.
    *   [ ] Create a Dockerfile for cross-compiling the Pi's ARM dependencies to avoid on-device build failures.

### Phase 2: Host Companion App Refactoring
**Objective:** Modularize the host CLI utility and ensure it clearly facilitates manual ID entry and secure synchronization.
*   **Prerequisites:** Python `curses`, existing `vault_manager.py`.
*   **Checklist:**
    *   [ ] Refactor `vault_manager.py` into distinct modules: `ui.py` (curses presentation), `storage.py` (JSON management), and `sync.py` (decoding/verification).
    *   [ ] Implement HMAC signature verification in the sync decoder to guarantee layout integrity.
    *   [ ] Update the UI text to explicitly instruct the user to type the selected `ID` directly into the Pi's dedicated physical keyboard.

### Phase 3: Pi OS Hardening & USB Gadget Initialization
**Objective:** Configure the DietPi OS as a headless, read-only system with hardware-toggled USB gadget modes (HID vs. Mass Storage).
*   **Prerequisites:** Raspberry Pi Zero 2 W, DietPi OS, physical GPIO slide switch, Micro-USB data cable.
*   **Checklist:**
    *   [ ] Disable all network interfaces (Wi-Fi, Ethernet overlays, Bluetooth stack).
    *   [ ] Implement a boot-time check script (`check_mode.sh`) to read a manual switch state. Early hardware will simply bridge a GPIO pin to enable Storage mode.
    *   [ ] Configure Default Mode A (Vault / HID): If the switch is open or fails, default to this mode. Initialize as HID Keyboard (`/dev/hidg0`). Mount the `vault.img` container with `ro,noexec,nosuid,nodev`. Set up `udev` rules for the unprivileged `vault_daemon` user.
    *   [ ] Configure Mode B (Sync / Mass Storage): If the GPIO pin is bridged, boot into read-write and expose `vault.img` via `libcomposite` as a USB Mass Storage device for database ingress. Document the procedure for users to back up their `.kdbx` file during this mode.
    *   [ ] Set up a `tmpfs` (RAM disk) mount and explicitly disable all OS swap space.

### Phase 4: Core Vault Daemon (Pi-Side)
**Objective:** Develop the core Python engine on the Pi for vault decryption, input listening, and secure HID injection.
*   **Prerequisites:** `pykeepass`, knowledge of Linux HID keycodes, dedicated physical keyboard.
*   **Checklist:**
    *   [ ] Implement the boot sequence: Wait for master password via the Pi keyboard. Perform Cryptographic Header Validation (verify `0x03D9A29A` and `0x67FB4BB5`) before parsing.
    *   [ ] Build the HID mapping engine with configurable micro-delays between keystrokes to prevent dropping characters. Include Caps Lock override macros.
    *   [ ] Implement the Input Listener with Duress PIN support (loading 'The Fake Vault').
    *   [ ] Build the injection logic: strictly use single `.get()` queries (no loops). Implement the two-stage flow (URL + Enter, wait for user, Username + Tab + Password + Enter).
    *   [ ] Integrate rate-limiting (e.g., token bucket) and the Honey-Pot trigger (wipe RAM/reboot on >3 requests in 30s or sequential lookups).
    *   [ ] Implement the Sync Exporter: Read entries, strip passwords, generate HMAC signature, compress via LZMA, encode to Base64, and transmit via HID.

### Phase 5: End-to-End Testing & Security Audit
**Objective:** Validate the system's resilience, data transmission reliability, and strict adherence to security constraints.
*   **Prerequisites:** Fully assembled hardware setup, test `.kdbx` database.
*   **Checklist:**
    *   [ ] Perform a **Transmission Test**: Sync a vault with 100+ entries and verify perfect decoding and HMAC validation on the host app.
    *   [ ] Perform an **Injection Test**: Verify exact password accuracy against a local text file, ensuring micro-delays prevent dropped characters.
    *   [ ] Perform a **Power-Loss Test**: Yank the power cord mid-operation. Verify read-only filesystem integrity and confirm no residual keys exist on the SD card upon reboot.
    *   [ ] Perform a **Volatility & Privilege Audit**: Use `htop` to confirm swap is disabled and the vault exists only in RAM. Verify the daemon runs entirely under the `vault_daemon` unprivileged user.

### Phase 6: Stretch Goals (Hardware Expansions)
**Objective:** Enhance the standalone physical capabilities and user experience of the Pi hardware.
*   **Prerequisites:** Secondary hardware modules, compatible E-Ink or OLED SPI display module.
*   **Checklist:**
    *   [ ] **Physical Display Integration:** Wire the display to the Pi's GPIO pins to show system status ("Vault Locked", "Ready for ID") once the final form factor of the case is selected.
    *   [ ] **OTP Display:** Add functionality to parse TOTP tokens from the KeePass database and display them on the physical screen for manual entry.
    *   [ ] **Encrypted Boot:** Configure Full Disk Encryption (LUKS) to protect the underlying OS against physical tampering.
